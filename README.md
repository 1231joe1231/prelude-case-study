# Prelude Case Study — Lead Intelligence Agent

A Chinese factory opens Prelude on Monday morning and wants to know which US importers to chase this week. This system reads the three data sources Prelude already has, scores every lead, and writes a 1–2 sentence rationale for the top picks — grounded in the actual data, not vibes.

## What it does

1. **Loads** the three CSVs (importers, their contacts, competing Chinese exporters) into a database.
2. **Builds** a "factory profile" — the product space the operator plays in — by reading what the competitor set sells.
3. **Scores** every lead from 0 to 1 against that profile, using six signals (formula below).
4. **Writes** a short rationale for the top-50 leads, citing specific HS codes, ports, supplier names, and contacts from the data.
5. **Shows** the result as a ranked table the operator can check off for outreach.

The score and ranking are deterministic — same inputs, same answer every time. Only the rationale text is generated by an LLM (Claude Haiku 4.5), and that text is verified against the source data before it reaches the operator.

## What the operator sees

| Tab | Purpose |
|---|---|
| **Deliverable** | The ranked outreach list. Each row: score (0–1), company, score breakdown by signal, reasoning, "selected for outreach" checkbox. |
| **Graph** | The shipping graph — which Chinese factories ship to which US importers. Hover edges for TEU / volume / supplier share. |
| **Tables** | The raw data behind every score. For debugging and trust-but-verify. |
| **Pipeline** | The "Run pipeline" button. Shows progress and per-rationale audit trail. |

The operator clicks **Run pipeline** to compute everything. The Deliverable table fills in within ~10 seconds for 50 leads. Rationales appear with placeholders first and upgrade to LLM text as each one finishes.

## How a lead gets ranked

Every lead is scored from 0 to 1 by combining six signals. The formula:

```
score  =  0.30 × volume
        + 0.35 × hs_fit
        + 0.25 × reachability
        + 0.10 × seniority
        + 0.00 × recency        ← see note
        + 0.00 × competitive    ← see note

then multiplied by:
        × 0.5    if the lead's HS codes don't overlap the factory profile at all
        × 0.4    if the lead is marked not_interested
        × 1.0    if the lead is marked synced_to_crm  (default state — no penalty)
```

Each signal is normalized to 0–1 first:

| Signal | What it measures | Formula |
|---|---|---|
| **volume** | how active an importer this is | log-scaled `total_shipments`, saturating at 500 |
| **hs_fit** | does the lead buy what the factory makes? | overlap between lead's HS codes and the inferred factory profile, as a ratio |
| **reachability** | can we reach this lead at all? | half-credit for any email + half-credit scaling with contact count |
| **seniority** | do we have a decision-maker? | 1.0 if a contact title matches CEO / VP / Owner / Director / Procurement, else 0.3 |
| **recency** | freshness of last shipment | exponential decay over 180 days *(weight currently 0 — see below)* |
| **competitive** | how crowded is this lead? | inverse of distinct competing exporters *(weight currently 0 — see below)* |

**Why recency and competitive are zero-weighted on the current data**: only 16 of 121 leads carry a usable `most_recent_shipment`, and only 5 of the top 50 have any resolved competitor edges. Including them as-is rewarded leads for *missing data*. Once coverage improves, weights revert to non-zero. The components are still computed and shown to the operator in the score breakdown.

**Status semantics**:
- `synced_to_crm` is the *default* in this dataset — every active lead is already in the CRM. It is not a negative signal and the rationale never criticizes it.
- `not_interested` means the operator contacted this lead before and they declined. It is not a disqualifier — strong new signal (fresh volume, a new senior contact, an HS shift) justifies a second attempt. The score takes a soft 0.4× penalty so high-signal declined leads can still surface above weaker fresh ones; the rationale is framed as a re-engagement angle.

## How each data source contributes

| Source | What we extract | Where it lands |
|---|---|---|
| **Leads CSV** (US importers' shipment data) | volume, recency, HS codes the importer buys, top destination ports, workflow status | volume, recency, hs_fit signals |
| **Personnel CSV** (contacts at importers) | does the lead have any email at all, is any contact senior (CEO / VP / Owner / etc.) | reachability, seniority signals |
| **Competitors CSV** (Chinese factories selling the same goods) | top HS codes across competitors → the factory's product profile; which competitors already supply each importer; supplier share % | hs_fit (via the profile) and competitive signal |

The three sources also share one hidden structure: a **shipping graph**. The competitors-to-customers list inside the competitor data and the suppliers list inside the lead data are the same edges viewed from opposite sides. The system merges them into one graph, lets us answer "is this lead already buying from a known competitor?" in one query, and renders the result on the Graph tab.

## Scaling from 121 leads to 100,000

- **Ingest and storage** are linear and run only at startup. At 100K leads, ingest takes minutes, not the operator's session.
- **Scoring is constant per lead and trivially parallelizable** — a few microseconds of arithmetic each. At 100K it remains negligible. The only thing that needs to change is to push the sort into the database (`ORDER BY score DESC LIMIT 50`) rather than reading everything into memory.
- **LLM cost scales with the top-N shown to the operator, not the dataset.** Whether there are 120 leads or 100,000, we generate ~50 rationales per refresh. With prompt caching, the per-rationale cost is small (~95% of input tokens are cached after the first call); a full refresh is under a penny on Haiku 4.5.
- The **operator's experience stays the same** at 100K — the table fills in within ~10 seconds because the LLM only runs on what's shown.

The architecture is designed so the LLM is never on the critical scaling path. The deterministic ranker chooses *who* — the LLM only writes *why*.

## Evaluating quality

**Built**

- **Factuality check on every rationale.** Before any LLM-generated text reaches the operator, it's validated: every cited HS code must appear in the lead's actual HS codes; every company-shaped phrase must match the lead's real name (catches subtle hallucinations like "Pearhead Inc" → "Pearson Inc"); at least one specific fact must be cited. Failures are silently replaced with a deterministic fallback and flagged in the audit trail.
- **Per-lead audit trail.** For any lead, the system exposes the exact prompt sent to the LLM, the raw response, what passed/failed the factuality check, and what the operator ultimately saw. A reviewer can answer "where did this claim come from" for every word in every rationale.
- **Golden dataset switch.** A small hand-crafted CSV set can be swapped in via `INPUT_VERSION=golden`, letting eval tests assert exact expected rankings (the curated golden CSVs are the next-phase deliverable).

**Planned next**

- **Ablation:** zero each weight, re-rank, measure correlation against the full ranking. Any weight whose ablation changes nothing is dead weight and should be retired.
- **Sanity-set eval:** hand-label ~10 leads as obviously-high / obviously-low, assert they land in the expected band of the ranking.
- **Factuality regression:** run the LLM on a fixed feature set N times, fail the build if hallucination rate >5%.

## Closing the feedback loop

The brief asks how operator feedback (the `status` column) closes the loop. The pieces:

1. **Capture every selection.** When the operator ticks "selected for outreach," persist that decision alongside the feature snapshot the lead had at the moment of decision. These are supervised labels in disguise.
2. **Re-fit weights from labels.** Once ~200 selections accumulate, fit a simple regression over the six signal scores against whether the lead was selected within 7 days. The new weights replace the hand-set ones; old and new can be A/B'd by version.
3. **Watch for drift.** When the factuality failure rate climbs or the rank-vs-selection correlation drops, alert. The loop closes itself.

The pieces are in place to do this — selections happen in the UI, every feature is preserved, weights are a single config value. The retrain itself is out of scope for this take-home.

## Working assumptions (called out per brief)

- **The factory's product profile is inferred, not given.** The brief doesn't pin a factory identity, so the profile is derived from the top HS codes in the competitor set — those 30 Chinese factories all ship the same category as our hypothetical operator. In production this would be a per-tenant input.
- **"Worth chasing this week"** means active recent shipping + product fit + reachable contact. `not_interested` leads still surface if the new signal is strong enough.
- **Single product vertical.** The dataset is curated around Christmas decorations / artificial trees. No cross-vertical leakage modeled.
- **Legacy `score` columns** in `leads.csv` and `personnel.csv` are ignored per the brief.
- **`synced_to_crm`** is the baseline state in this dataset (every eligible lead carries it), so penalizing it would suppress everyone. Treated as neutral.
- **`not_interested`** = previously declined, not dead. Soft 0.4× penalty, not a filter.
- **The rationale cache** lives in memory and is cleared on restart. The brief asks for no persistence beyond the demo; this satisfies it.

## Stack and run

**Backend**: FastAPI + SQLite + Anthropic SDK (port 8000).
**Frontend**: Vite + React + Tailwind (port 5173).

```powershell
# Backend
cd backend
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
copy .env.example .env       # paste ANTHROPIC_API_KEY into .env (optional)
uvicorn app.main:app --port 8000

# Frontend (new shell)
cd frontend
npm install
npm run dev
```

Open <http://localhost:5173>. Backend ingests CSVs at startup. Click **Run pipeline** on the Pipeline tab to compute rankings and rationales. Without an API key, the rationales are generated by a deterministic fallback that still cites literal facts — the demo runs without a key.
